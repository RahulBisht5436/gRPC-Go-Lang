# `proto/client.pb.go` — Generated message structs

> ⚠️ Iss file ko **kabhi haath se mat edit karo**. File ki pehli line khud kehti hai:
> ```go
> // Code generated by protoc-gen-go. DO NOT EDIT.
> ```
> `protoc` regenerate karega aur manual changes lost ho jaayenge.

## Ye file kya hai?

`protoc-gen-go` plugin ne `client.proto` ke saare **`message`** declarations ko Go code me convert kiya. Yaha kuch business logic nahi hai — bas data structures (structs) aur unke helpers.

`service`-related code alag file me jaata hai: [`client_grpc.pb.go`](./client_grpc.pb.go.md) — vo `protoc-gen-go-grpc` plugin banata hai.

> **Streaming context me note**: Iss project me service `SendUser` (client streaming) hai. Lekin **message structs is se totally untouched** rehte hain — `UsersRequest` aur `UserResponse` ki shape unary, server-streaming, client-streaming, bidirectional — sab modes me ek hi hoti hai. Streaming sirf "kitne messages travel karte hain wire pe" decide karta hai, "har message ka shape kya hai" wo to message definition se aata hai. Yahi reason hai ki ye file har streaming mode me **almost identical** dikhti hai — sirf descriptor bytes me ek flag badalta hai.

## Struct fields ko samjho

```go
type UsersRequest struct {
    state         protoimpl.MessageState   // protobuf internal
    Name          string                   // <-- tumhara field
    unknownFields protoimpl.UnknownFields  // protobuf internal
    sizeCache     protoimpl.SizeCache      // protobuf internal
}
```

- **`Name`** — yahi tumhare proto wala field hai. Capital `N` se start hota hai (Go me **exported** hone ke liye PascalCase chahiye).
- Baaki 3 fields (`state`, `unknownFields`, `sizeCache`) **protobuf runtime ke internals** hain. Inko ignore karo.

### Struct tag dekho

```go
Name string `protobuf:"bytes,1,opt,name=name,proto3" json:"name,omitempty"`
```

Ye **struct tag** hai (backticks me). Iska decode:

| Part | Matlab |
|---|---|
| `protobuf:"..."` | Protobuf encoder/decoder ke liye instructions |
| `bytes` | Wire type (length-prefixed bytes — strings ke liye) |
| `1` | Field number (proto se: `string name = 1`) |
| `opt` | Optional (proto3 default) |
| `name=name` | Original proto name |
| `proto3` | Proto syntax version |
| `json:"name,omitempty"` | JSON serialization ke liye (gRPC-Web ya REST-via-grpc-gateway) |

### `UserResponse` me ek interesting cheez

```go
Result string `protobuf:"bytes,1,opt,name=Result,proto3" json:"Result,omitempty"`
```

Notice — tag me `name=Result` aur JSON tag me `"Result"` (capital R). Ye isliye hai ki tumne proto me bhi capital `Result` likha tha (`string Result = 1;`). Agar tum convention follow karte (`string result = 1;`), to JSON tag `"result"` hota — REST gateways aur web clients ke liye chhota difference banata hai.

## Helper methods generate hue

Har message ke saath ye methods automatic banti hain:

### 1. `Reset()` — message ko zero value pe le aata hai

```33:38:proto/client.pb.go
func (x *UsersRequest) Reset() {
	*x = UsersRequest{}
	mi := &file_client_proto_msgTypes[0]
	ms := protoimpl.X.MessageStateOf(protoimpl.Pointer(x))
	ms.StoreMessageInfo(mi)
}
```

Pool-based memory reuse ke liye useful. Tum directly call nahi karoge mostly.

> **Streaming relevance**: Client streaming me client `Send()` har bar nayi `&pb.UsersRequest{...}` literal banata hai (jaisa tumhare `client/main.go` ke loop me hai), to `Reset()` directly istemaal nahi hota — but agar tum performance-tune karke object pool use karoge (e.g., 10k requests/sec wala upload service), to ye method kaam aayega.

### 2. `String()` — debug format

```40:42:proto/client.pb.go
func (x *UsersRequest) String() string {
	return protoimpl.X.MessageStringOf(x)
}
```

`fmt.Println(req)` ya `log.Printf("%v", req)` jab use karte ho, isi ka output dikhta hai. Output dikhta hai `name:"Rahul Bisht"` jaisa.

### 3. `ProtoMessage()` — marker method

```44:44:proto/client.pb.go
func (*UsersRequest) ProtoMessage() {}
```

Bas ek **empty marker** method. Ye kahin call nahi hota — sirf type system ko bata raha hai "ye proto message hai". Interface satisfaction ke liye.

### 4. `ProtoReflect()` — reflection API

```46:56:proto/client.pb.go
func (x *UsersRequest) ProtoReflect() protoreflect.Message {
	mi := &file_client_proto_msgTypes[0]
	if x != nil {
		ms := protoimpl.X.MessageStateOf(protoimpl.Pointer(x))
		if ms.LoadMessageInfo() == nil {
			ms.StoreMessageInfo(mi)
		}
		return ms
	}
	return mi.MessageOf(x)
}
```

Advanced — protobuf reflection ke liye. Tools jaise `grpcurl`, debug proxies, schema-based code is method ka use karte hain. Tumhare normal handler code me iska direct istemaal nahi.

### 5. `GetXxx()` — nil-safe getters (sabse important!)

```63:68:proto/client.pb.go
func (x *UsersRequest) GetName() string {
	if x != nil {
		return x.Name
	}
	return ""
}
```

```107:112:proto/client.pb.go
func (x *UserResponse) GetResult() string {
	if x != nil {
		return x.Result
	}
	return ""
}
```

Yeh **important** hai. Tumhare handler me always `req.GetName()` use karo, **direct `req.Name` nahi** (tumhare current `server/main.go` me already `req.GetName()` use ho raha hai — good!). Client side me bhi tumne `resp.GetResult()` use kiya hai. Reason:

```go
// In nil-able pointer
var req *pb.UsersRequest = nil

req.Name       // ❌ panic: nil pointer dereference
req.GetName()  // ✅ returns "" safely
```

#### `resp.GetResult()` ka special importance iss project me

Tumhare `client/main.go` me ye sequence hai:

```go
resp, err := stream.CloseAndRecv()
if err != nil {
    log.Printf("Failed Request : %v", err)
}

fmt.Println("...", resp.GetResult())
```

Agar `CloseAndRecv` fail hua to `resp = nil`. Lekin agle line me `resp.GetResult()` chala — **and ye crash NAHI karega** kyunki `GetResult()` nil-safe hai. Empty string return karega. Yahi getter ka real-world value.

> Better pattern: error hone pe `log.Fatalf` ya `return` kar do — bekaar processing avoid karne ke liye. Detail [client-main.go.md](./client-main.go.md) me.

## File ke neeche wala "magic" code

```116:124:proto/client.pb.go
const file_client_proto_rawDesc = "" +
	"\n" +
	"\fclient.proto\x12\fclientStream\"\"\n" +
	"\fUsersRequest\x12\x12\n" +
	"\x04name\x18\x01 \x01(\tR\x04name\"&\n" +
	"\fUserResponse\x12\x16\n" +
	"\x06Result\x18\x01 \x01(\tR\x06Result2S\n" +
	"\vUserService\x12D\n" +
	"\bSendUser\x12\x1a.clientStream.UsersRequest\x1a\x1a.clientStream.UserResponse(\x01B Z\x1eexample.com/clientStream/protob\x06proto3"
```

Ye **raw protobuf descriptor** hai — tumhare proto file ka encoded form. Reflection, dynamic decoding, aur `grpcurl` jaise tools is descriptor ka istemaal karte hain.

Pad nahi sakte directly (binary jaisa hai), but agar dhyan se dekho to `UsersRequest`, `name`, `UserResponse`, `Result`, `UserService`, `SendUser`, `example.com/clientStream/proto` — sab dikh rahe hain.

> **Spot the streaming flag**: Notice `(\x01` byte (request type ke baad). Yahi vo flag hai jo descriptor me batata hai "ye method ki **input streaming** hai" (yaani client side se stream aa raha hai). Server-streaming version me ye `(` (open paren) ke baad `0\x01` byte hota tha (output streaming). Yaani descriptor me bhi mode encode hota hai — ek-ek byte se mode ka pata chal sakta hai.

## Beginner ke liye yaad rakhne wali baatein

| Sawaal | Jawab |
|---|---|
| Ye file mai edit kar sakta hu? | **Nahi.** `protoc` regenerate karega aur changes mit jaayenge. |
| `name` proto me, `Name` Go me — kyu? | Go me exported field PascalCase me hote hain. `protoc` automatic convert karta hai. |
| `GetName()` ya `Name` — kya use karu? | Hamesha `GetName()` — nil-safe hai. |
| `Result` (capital R proto me) — sahi hai? | Kaam karta hai, but proto convention `snake_case` (`result`). Beginner-level: ignore karo. Production code: rename karo. |
| `protoimpl.MessageState`, `unknownFields`, `sizeCache` kya hai? | Protobuf runtime internals. Ignore karo. |
| Ye sab generate kab hota hai? | Jab tum `protoc --go_out=. client.proto` chalao. |

## Proto change to flow kya hai?

```
1. Edit client.proto (e.g., add `int32 age = 2;` to UsersRequest)
2. cd "gRPC project/Greet Client Stream"
3. protoc ... client.proto    <-- ye client.pb.go ko regenerate karega
4. Ab Go code me `req.GetAge()` available hoga
5. Server/client code me usage update karo (server me Recv loop me use, client me Send call me populate)
```

Bilkul iterative loop hai — proto edit, regenerate, code update.

### Streaming-specific note

Agar tum proto me server streaming se client streaming pe switch karte ho (`returns (stream X)` → `(stream X) returns`):

- **Iss file me** kuch nahi badlega visibly — message structs same.
- `file_client_proto_rawDesc` ke andar ek-do bytes badlenge (streaming direction flag).
- Sara badlaav [`client_grpc.pb.go`](./client_grpc.pb.go.md) me hota hai — vahi service interfaces aur dispatch table rakhta hai.

Yaani: **messages ki file streaming mode se na ke barabar badalti hai**. Yahi reason hai ki ye doc unary aur server streaming version ke `pb.go.md` se 90% same hai — actual structs ki kahani modes ke beech identical hai.
