## Ye file kya hai?

`protoc-gen-go` plugin ne `bideirectionalStreams.proto` ke saare **`message`** declarations ko Go code me convert kiya. Yaha kuch business logic nahi hai — bas data structures (structs) aur unke helpers.

`service`-related code alag file me jaata hai: [`bideirectionalStreams_grpc.pb.go`](./bideirectionalStreams_grpc.pb.go.md) — vo `protoc-gen-go-grpc` plugin banata hai.

> **Bidirectional streaming context me note**: Iss project me service `GreetEveryone` (bidirectional) hai. Lekin **message structs is se totally untouched** rehte hain — `GreetRequest` aur `GreetResponse` ki shape unary, server-streaming, client-streaming, bidirectional — sab modes me ek hi hoti hai. Streaming sirf "kitne messages travel karte hain wire pe aur kis direction me" decide karta hai, "har message ka shape kya hai" wo to message definition se aata hai.

## Struct fields ko samjho

```go
type GreetRequest struct {
    state         protoimpl.MessageState   // protobuf internal
    FirstName     string                   // <-- tumhara field
    unknownFields protoimpl.UnknownFields  // protobuf internal
    sizeCache     protoimpl.SizeCache      // protobuf internal
}
```

- **`FirstName`** — yahi tumhare proto wala field hai. Capital `F` se start hota hai (Go me **exported** hone ke liye PascalCase chahiye).
- Baaki 3 fields (`state`, `unknownFields`, `sizeCache`) **protobuf runtime ke internals** hain. Inko ignore karo.

### Struct tag dekho — `snake_case` ka **fayda** dikh raha

```go
FirstName string `protobuf:"bytes,1,opt,name=first_name,json=firstName,proto3" json:"first_name,omitempty"`
```

Notice **`json=firstName`** aur **`json:"first_name,omitempty"`** — dono encoded hain because tumne `snake_case` use kiya proto me. Decode:

| Part | Matlab |
|---|---|
| `protobuf:"..."` | Protobuf encoder/decoder ke liye instructions |
| `bytes` | Wire type (length-prefixed bytes — strings ke liye) |
| `1` | Field number (proto se: `string first_name = 1`) |
| `opt` | Optional (proto3 default) |
| `name=first_name` | Original proto name (snake_case) |
| `json=firstName` | **Proto-level JSON name** — camelCase (proto3 default mapping) |
| `proto3` | Proto syntax version |
| `json:"first_name,omitempty"` | **Go-level JSON tag** — snake_case (matches proto name exactly) |

Tumhare pichle projects me (`firstName` camelCase wale) ye tag aisa nahi tha — sirf `firstName` had been preserved. snake_case se zyada explicit encoding milta hai REST gateways ke liye.

Pool-based memory reuse ke liye useful. Tum directly call nahi karoge mostly.

> **Bidirectional streaming relevance**: Yahaan dono client aur server `Send()` ke samay nayi struct literals banate hain (jaisa tumhare server me `&pb.GreetResponse{Result: respondMessage}` aur client me `&pb.GreetRequest{FirstName: name}`). To `Reset()` directly istemaal nahi hota — but agar tum high-throughput chat server bana rahe ho (millions of messages/sec), `sync.Pool` ke saath `Reset()` use karke allocation cost kam kar sakte ho.

### 5. `GetXxx()` — nil-safe getters (sabse important!)

```61:66:proto/bideirectionalStreams.pb.go

	if x != nil {
		return x.FirstName
	}
	return ""
}
```

```105:110:proto/bideirectionalStreams.pb.go
func (x *GreetResponse) GetResult() string {
	if x != nil {
		return x.Result
	}
	return ""
}
```

Yeh **important** hai. Tumhare handler me always `req.GetFirstName()` use karo, **direct `req.FirstName` nahi**. Tumhare current code me dono jagah sahi getters use ho rahe hain:

- Server: `req.GetFirstName()` ✓
- Client: `res.GetResult()` ✓

Reason:

```go
var req *pb.GreetRequest = nil

req.FirstName       // ❌ panic: nil pointer dereference
req.GetFirstName()  // ✅ returns "" safely
```

#### Bidirectional ka specific concern

Bidirectional me `res` nil ho sakta hai jab `Recv` error return kare (ya stream broken hai). Tumhare current code me ye scenario hai:

```go
res, err := stream.Recv()
if err == io.EOF { close(waitc); return }
if err != nil { log.Fatalf("Recv error : %v", err) }
log.Printf("Response: %s", res.GetResult())   // <-- safe due to nil-safe getter
```

Lekin yahaan `log.Fatalf` already program exit karega agar err non-nil. To `res.GetResult()` sirf success case me chalega — fir bhi getter ka habit accha hai.

## TL;DR table

| Cheez | Matlab |
|---|---|
| `type GreetRequest struct { FirstName string }` | Proto se Go ka 1-to-1 mapping |
| Field tag (`protobuf:"..." json:"..."`) | Encoder/decoder + JSON serialization instructions |
| `Reset()`, `String()`, `ProtoMessage()`, `ProtoReflect()` | Protobuf runtime helpers (rarely directly used) |
| `GetXxx()` | Nil-safe getters — **always use these** |
| `rawDesc` constant | Encoded version of proto file for reflection |
| `init()` | Auto-registers descriptor on package import |
| Bidirectional flag in descriptor | Both `(\x01` and `0\x01` bytes encoded |

> **Bidirectional bonus**: ye file unary, server stream, client stream, aur bidirectional sab versions me **almost identical** rahegi — sirf rawDesc bytes me streaming flags badalte hain. Yahi message-vs-service separation ka fayda hai.
